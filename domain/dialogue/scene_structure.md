# Dialogue Session — File Structure

All files for a dialogue session live under `infrastructure/dialogues/{dialogue_id}/`.

A scene supports **N participants** (currently 2 or 3). Throughout these files, characters are referenced by their **directory id** (e.g. `alice-smith`) — the directory name under `infrastructure/characters/`. The schema is participant-id-keyed everywhere; there are no slot names like `charA`/`charB`.

---

## `characters.json`

Participants in this session. Written by the frontend at scene setup, updated after leading character selection.

```json
{
  "participants": {
    "<char_id>": {
      "id": "string — character directory name (e.g. alice-smith)",
      "name": "string — display name",
      "data_path": "string — relative path to data.json (e.g. infrastructure/characters/alice-smith/data.json)"
    }
  },
  "leading_id": "string — set after user selects leading character",
  "player_id": "string | null — set when TAVERN_VERBATIM=on (player character mode)"
}
```

- `participants` is a dict keyed by `char_id`. Insertion order is meaningful: it defines the default left-side display order in the chat panel.
- `leading_id` is absent until stage 2 (dialogue initialization). It must match a key in `participants`.
- `player_id` is absent or null in director mode (`TAVERN_VERBATIM=off`). When `TAVERN_VERBATIM=on`, it identifies which participant is controlled by the user. The player character's dialogue is typed in the UI and written directly to the chat — the AI never generates turns for this character (they are excluded from the planner's speaking set). `player_id` must match a key in `participants` when set.
- For raw characters pending repack, the participant entry has `needs_repack: true` and `raw_path` instead of `data_path`. The `id` is pre-derived from the character name via the same `sanitizeName` rule the frontend uses (`name.toLowerCase().replace(/[^a-z0-9]+/g, '_').replace(/^_|_$/g, '')`), so the id is stable from the moment the scene is created.

## `scenario.json`

Generated by `optimize_scenario`. Contains each character's adapted scenario and opening line options.

```json
{
  "dialogue_id": "string — UUID",
  "generated_at": "string — ISO 8601 timestamp",
  "participants": {
    "<char_id>": {
      "name": "string — display name",
      "scenario": "string — this character's typical_scenario adapted for the other participants",
      "openings": [
        "string — greeting adapted for the other participants",
        "string — alternate_greetings[0] adapted for the other participants"
      ]
    }
  }
}
```

Every participant has a scenario and opening list regardless of which becomes the leading character — that selection happens in the UI after this file is written.

---

## `context_cache.json` + `context_cache_{char_id}.json`

Pre-built per-session character data cache. Written by `build_context_cache.py` in Phase 0. Splits into a meta file and per-character files:

**`context_cache.json`** — meta only:

```json
{
  "scenario": "string — leading character's adapted scenario text",
  "leading_id": "string — leading character id",
  "participant_ids": ["string — sorted array of all participant ids"]
}
```

**`context_cache_{char_id}.json`** — one file per participant:

```json
{
  "identity": { "full_name": "...", "gender": "..." },
  "appearance": { "summary": "...", "height": "...", "build": "...", "typical_clothing": "..." },
  "personality": { "core_traits": [], "emotional_baseline": "...", "quirks": [] },
  "speech": { "voice_description": "...", "vocabulary_level": "...", "speech_patterns": [], "sample_lines": [] },
  "behavior": { "social_style": "...", "relationship_defaults": "...", "triggers": "...", "stress_behaviors": "..." }
}
```

- String fields are truncated to 500 chars and list fields to 5 items to keep reads compact.
- Lorebook entries are NOT in these files — they live in `active_lorebook.json` (rebuilt every turn).
- The planner reads the meta + all per-character files. The validator and turn agents do NOT read these — they use `character_briefs` embedded in `reply_plan.json` instead.
- Mtime-skipped: if no source data has changed since the cache was last built, `build_context_cache.py` skips the rebuild.

## `goals.json` (narrator mode only)

Scene objectives for narrator-mode dialogues. Replaces `scenario.json` as the scene anchor when `TAVERN_CHAT_MODE=narrator`. Written by the frontend at scene setup.

```json
{
  "dialogue_id": "string — UUID",
  "scene": "string — setting description (location, time, atmosphere, who is present)",
  "goals": [
    {
      "id": "string — unique goal identifier (e.g. recruit_elara)",
      "description": "string — what the conversation is trying to accomplish",
      "priority": "main | secondary",
      "resolutions": {
        "success": "string — what happens if the goal succeeds",
        "player_refuses": "string — what happens if the player walks away",
        "npc_refuses": "string — what happens if the NPC declines",
        "conditional": "string — partial success with conditions (optional)"
      },
      "status": "active | resolved",
      "resolved_as": "string | null — set by planner when goal resolves (e.g. 'success', 'npc_refuses')"
    }
  ]
}
```

- The planner focuses on the `main` priority goal; `secondary` goals inform NPC behavior but don't drive the conversation.
- When the planner determines a goal has naturally resolved through the conversation, it sets `resolved_as` and `status: resolved` in its plan output. The orchestrator writes the update back to `goals.json`.
- NPCs initiate the conversation based on their personality, the scene setting, and the goals. There are no pre-generated greetings — the planner decides who speaks first and what they open with.
- Multiple goals can be active simultaneously but the planner focuses on one at a time.

---

## `complete.json` (narrator mode only)

Written by the orchestrator when the planner resolves a goal and sets `dialogue_complete: true`. Presence of this file means the dialogue is closed — no further `generate_reply` tasks should be processed for this dialogue.

```json
{
  "completed_at": "string — ISO 8601 timestamp",
  "goal_id": "string — which goal resolved",
  "outcome": "string — resolution type (success, npc_refuses, player_refuses, conditional, custom)",
  "detail": "string — planner's description of what happened"
}
```

The UI reads this file to display a completion banner and disable all input controls.

---

## `active_lorebook.json`

Per-turn filtered lorebook. Written by `build_active_lorebook.py` in Phase 0 every turn. Dict-keyed by char_id so agents read only their own character's slice.

```json
{
  "generated_at": "string — ISO 8601",
  "haystack_char_count": 0,
  "entries_by_char": {
    "<char_id>": [
      {
        "keys": ["string — original lorebook keys"],
        "content": "string — lorebook entry content",
        "priority": 0,
        "trigger": "always | pattern | cross_char | keyword",
        "matched": ["string — what caused the match (debug)"]
      }
    ]
  }
}
```

- Entries sorted by priority descending within each character's slice.
- Trigger types: `always` (entry.always=true), `pattern` (baseline key pattern match), `cross_char` (other participant's name in entry keys), `keyword` (entry key found in recent prose haystack).

---

## `full_chat.json`

Complete dialogue history. Grows unboundedly. Never trimmed.

```json
[
  { "speaker": "char_id", "text": "string — reply with \\n\\n paragraph breaks" },
  { "speaker": "char_id", "text": "string" }
]
```

`speaker` is always a participant id from `characters.json → participants`.

---

## `recent_chat.json`

Sliding context window used by agents. Always trimmed to the last **10 entries** after each append.

Same shape as `full_chat.json`. Agents read this, never `full_chat.json`, to bound context cost.

---

## `reply_plan.json`

Round plan written by the planner agent. Variable-length `turns[]` array — the planner picks who speaks each round (1 to N participants), and the order they speak in. Uses a structured `beats` array per turn instead of free-prose summaries.

```json
{
  "turn_order": ["<char_id_1>", "<char_id_2>", "..."],
  "round_protagonist": "<char_id>",
  "turns": [
    {
      "speaker": "<char_id>",
      "weight": "reaction | action | inflection | climax",
      "beats": ["one atomic beat ≤25 words", "another beat"],
      "tone": "short mood/voice gloss ≤40 words",
      "direction_applied": "<excerpt from user_prompt or null>",
      "rule_triggers": [],
      "voice_notes": "...",
      "ends_tbc": false,
      "tbc_state": null
    }
  ],
  "scene_context_summary": "1-2 sentence snapshot",
  "pending_tbc": null,
  "character_briefs": {
    "<char_id>": {
      "name": "...", "gender": "...", "height": "...", "build": "...",
      "appearance_summary": "...", "core_traits": [], "emotional_baseline": "...",
      "quirks": [], "voice_description": "...", "speech_patterns": [], "vocabulary_level": "..."
    }
  }
}
```

- `turn_order[i]` must equal `turns[i].speaker` for every `i`.
- `round_protagonist` identifies the character driving the round's beat.
- `beats` array length is hard-capped by weight: reaction 1-2, action 2-3, inflection 3-4, climax 4-6.
- `character_briefs` contains full cross-reference data (appearance + personality + voice) so the validator and turn agents don't need to read context_cache files.
- `pending_tbc` is set when the last turn ends on a "to be continued" freeze — `merge_reply.py` reads this to write `tbc.json` for the next round.
- Not every participant needs to speak in every round. A round may be a 1-character monologue, a 2-character exchange, or a full N-character round, as the scene dynamics demand.

---

## `reply_turn_{i}.json`

Per-turn prose output written by `generate_reply_turn` agents. One file per entry in `reply_plan.turns[]`, indexed by position (`reply_turn_0.json`, `reply_turn_1.json`, …). Schema:

```json
{
  "speaker": "<char_id>",
  "text": "<full prose with \\n\\n paragraph breaks>"
}
```

Materialized verbatim turns (`turns[0].verbatim == true`) are written by `apply_verbatim.py` directly. All other turn files are written by spawned agents reading `turns[i]` from the plan plus all prior turn files (`reply_turn_0.json` … `reply_turn_{i-1}.json`) for cross-turn reaction context.

`merge_reply.py` reads `reply_plan.turns[]` in order, picks up the matching `reply_turn_{i}.json` for each entry, and assembles the final `pending_turns.json`.

---

## `memory.json` _(optional)_

General scene memory written by agents when notable facts accumulate. Read by `generate_reply` if present.

No fixed schema — free-form JSON object. Not currently generated by any implemented agent; reserved for future use.

---

## Path conventions

- Character `id` in `speaker` fields and `participants` keys is the **directory name** (e.g. `alice-smith`), not `meta.id` inside `data.json`.
- `data_path` values are relative to the project root: `infrastructure/characters/{id}/data.json`.
